00 Three levels, one job

Every language exists to close the same gap: human idea → machine action.

In Lecture 01 we saw the CPU only ever executes patterns of 1s and 0s. But no one writes programs that way — instead we've built three levels of language, each trading closeness-to-the-machine for ease-of-use-for-humans. Click a level to explore it.

01 Machine language

The only language the CPU actually understands.

Machine language is a series of binary patterns (0s and 1s) that directly correspond to specific operations the CPU can perform — exactly the "machine code" that came out of the abstraction stack in Lecture 01. Each instruction typically has an opcode (which operation) and an operand (what data). It is executed directly by the hardware with no translation needed — but it's extremely difficult for humans to read, write, or debug.

02 Assembly language

Giving binary opcodes short, human-readable names.

Assembly language replaces raw binary opcodes with short, memorable mnemonics — MOV, ADD, JMP — that correspond almost one-to-one with machine instructions. It's still tied closely to a specific CPU's design, but far more readable than binary. A translator called an assembler converts assembly code into machine code.

Interactive · assembler at work
The Assembler

Reads assembly source line by line and produces the exact machine code for that specific CPU family — a nearly 1-to-1 translation, so the programmer still needs to think close to the hardware (registers, memory addresses).

TAKEAWAY · Assembly = mnemonics for machine instructions, translated 1-to-1 by an assembler. Readable by humans, but still CPU-specific.
03 High-level languages

Write like a human. Let a translator worry about the machine.

High-level languages (Python, Java, C++, JavaScript…) use English-like keywords and structures, hide the details of any particular CPU, and let one line of code often stand in for many machine instructions. This makes programs faster to write, easier to read, and portable — the same source code can often run on very different machines, once translated for each.

04 Translating high-level code

Two ways to turn your code into something the CPU can run.

A high-level program must still become machine code eventually. Two common strategies do this:

Compiler

Translates the entire source program into machine code before it runs, producing a standalone executable file. Errors are caught all at once, before execution; the resulting program then runs fast on its own.

Interpreter

Translates and executes the program line by line, on the fly, with no separate executable produced. Easier for testing and interactive use, but generally slower since translation happens every time it runs.

TranslatesThe whole program, before runningLine by line, while running

TAKEAWAY · Compiler = translate everything first, then run fast. Interpreter = translate and run one line at a time, slower but flexible.

06 Why it matters

Every program you will ever write travels this same road down to voltage.

Understanding the three levels of language — and how compilers, interpreters and assemblers connect them — explains why some languages are faster, why "portable" code still needs translating for each machine, and why a single typo can be caught before running (compiled) or crash the program mid-way (interpreted). This is the foundation for every programming course that follows.

Choose the right tool

Need raw speed? A compiled language. Need to test quickly? An interpreted one.

Debug with context

Knowing whether your language compiles or interprets tells you when and how errors will appear.

Exam gold

"Differentiate machine, assembly and high-level languages" and "compiler vs interpreter" are classic questions — now covered in full.
